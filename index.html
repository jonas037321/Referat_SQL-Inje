Was ist SQL Injection?

SQL Injection (SQLi) ist eine der gefährlichsten Web-Sicherheitslücken. Angreifer nutzen sie, um:

  • Unberechtigt auf Datenbanken zuzugreifen
  • Sensible Daten zu stehlen (Passwörter, Kreditkarten, Personendaten)
  • Daten zu manipulieren oder zu löschen
  • Administratorzugriff zu erlangen
  • Malware einzuschleusen
OWASP Top 10: SQL Injection ist die #1 Web-Sicherheitslücke seit Jahren!

Das verwundbare Login

Typischer PHP-Code mit String-Konkatenation:

$user = $_POST['username']; $pass = $_POST['password']; $query = "SELECT * FROM users WHERE username = '$user' AND password = '$pass'"; $result = mysqli_query($conn, $query);

Das Problem: Benutzereingaben werden direkt in SQL eingebunden!

💡 Wenn der Benutzer ein ' (Hochkomma) eingibt, kann die SQL-Struktur zerstört werden!

Das Hochkomma – Der Schlüssel

Erwartete Query:
SELECT * FROM users
WHERE username = 'max'
AND password = 'pass123'
Mit Injection (Username = ' OR '1'='1):
SELECT * FROM users
WHERE username = '' OR '1'='1
AND password = 'pass123'

Ergebnis: Die WHERE-Bedingung ist IMMER WAHR → Alle User werden zurückgegeben!

Demo: Klassische SQL Injection

Login-Formular (VULNERABLE MODE)
SELECT * FROM users WHERE username = '' AND password = ''
💡 Probiere die Injection-Buttons aus oder gib selbst ' OR '1'='1' -- ein!

Verschiedene Injection-Varianten

  • ' OR '1'='1' --
    Klassische Methode: Macht die WHERE-Bedingung IMMER wahr
  • admin'--
    Comment-Bypass: Ignoriert das Passwort durch SQL-Kommentar
  • '; DROP TABLE users; --
    Destruktiv: Löscht ganze Tabellen
  • UNION SELECT ...
    Daten extrahieren: Holt verborgene Datenbankdaten
  • SLEEP() / TIME-BASED BLIND SQLi
    Blind Injection: Exfiltration durch Zeitmessungen
⚠️ Diese sind nur einige Beispiele. Die Möglichkeiten sind nahezu unbegrenzt!

Blind SQL Injection

Was wenn die Datenbank keine Fehler anzeigt?

  • Time-Based: Angreifer sendet Queries die bei bestimmten Bedingungen lange dauern
  • Boolean-Based: Unterschiedliche Responses (wahr/falsch)
  • Exfiltration: Daten werden Bit-für-Bit durch Messungen extrahiert
username = 'admin' AND SLEEP(5) --
// Wenn admin existiert → 5 Sekunden Verzögerung!

Dadurch können Angreifer jedes Passwort zeichenweise erraten!

Demo: Blind SQL Injection (Timing)

Zeitbasierte Datenexfiltration

Der Server antwortet langsam, wenn die Bedingung erfüllt ist. So können wir Daten extrahieren:

ANTWORTZEIT:
- ms
⏱️ Wenn die Response > 2000ms dauert, ist die Bedingung erfüllt!

UNION-Based SQL Injection

Mit UNION SELECT können mehrere Tabellen kombiniert werden:

// Injection:
username = ' UNION SELECT id, username, password FROM users --

// Resultierende Query:
SELECT * FROM users
WHERE username = '' UNION SELECT id, username, password FROM users --
AND password = ''

Angreifer sieht nun ALLE Benutzerdaten inklusive Passwörter!

💡 UNION ist besonders mächtig, weil Daten direkt im Response sichtbar sind.

Die Lösung: Prepared Statements

Prepared Statements trennen Code von Daten:

// SICHER: Prepared Statement (PHP mysqli)
$stmt = $conn->prepare("SELECT * FROM users WHERE username = ? AND password = ?");
$stmt->bind_param("ss", $user, $pass);
$stmt->execute();
$result = $stmt->get_result();
  • Das ? wird als Platzhalter behandelt
  • Benutzereingaben werden IMMER als Daten interpretiert
  • SQL-Metazeichen wie ' werden automatisch escaped
  • Injection ist unmöglich

Demo: Prepared Statements (SICHER)

Login mit Prepared Statements
PREPARE "SELECT * FROM users WHERE username = ? AND password = ?"
BIND param1, param2
✅ Hier funktioniert die Injection NICHT – egal was du eingibst!

Best Practices für sichere Applikationen

  • Prepared Statements / Parameterized Queries verwenden (immer!)
  • Input Validation: Daten validieren und filtern
  • Least Privilege: DB-User mit minimalen Rechten
  • Error Handling: Keine SQL-Fehler an User anzeigen
  • Web Application Firewall (WAF): Zusätzliche Schutzebene
  • Regular Updates: Framework und Dependencies aktuell halten
  • Security Testing: Penetration Tests durchführen
  • Code Review: Andere Entwickler Code überprüfen lassen
🛡️ Goldene Regel: Prepared Statements sind das Mittel #1!
