SQL INJECTION

Eine technische Präsentation über Web-Security

HTL Wirtschaftsinformatik • Sieberer & Zeindl

Einleitung

Metapher: Biometrisches Hochsicherheitsschloss mit Zettel „Ich bin der Admin“ überlistet.

  • SQL-Injection = kein technischer Fehler
  • Logischer Fehler im Code
  • Fatale Verwechslung von Inhalt und Befehl
Die Anwendung schmiedet die Falle für ihre eigene Zerstörung selbst.

Das Hochkomma

Der Star des Angriffs: Das einfache Hochkomma (').

  • Markiert in SQL Textwerte-Grenzen (Anfang/Ende)
  • Ungefilterte Benutzereingaben = Kontrollverlust über Parser
  • Injiziertes ( ' ): „Hier enden Daten, neuer Befehl folgt!“
Angreifer wird vom passiven Nutzer zum Co-Autor der Datenbankabfrage.

Risikobewertung

Dauerbrenner seit 20 Jahren:

  • OWASP Top 10: Gefährlichster Angriffsvektor
  • Durchschnittskosten Date Breach: 4 Millionen Dollar
  • Relationale Datenbanken & SQL als Hauptziel
  • Generalschlüssel zu wertvollen Firmen- & Kundendaten
Folge: Totaler Verlust der Datenhoheit.

Mechanik der Injektion

Server erwartet reine Daten, bekommt aber Code:

  • Eingabe ist ein syntaktisches Fragment
  • Verändert Befehlslogik zur Laufzeit
  • Input wird irrtümlich vertraut = Ausführung mit vollen Privilegien
Ein Angriff auf die semantische Integrität des Programms.

Der vulnerable Code

Bsp:

SELECT * FROM users WHERE username = ' + $_POST['user'] + ' AND password = ...
  • Input: admin' --
  • -- ignoriert Rest der Abfrage (Bypass Passwortprüfung)
  • Angreifer ist drin – ohne Passwort.
Login-Formular (VULNERABLE MODE)
SELECT * FROM users WHERE username = '' AND password = ''

Tautologien

Nutzen absolut wahrer Aussagen:

  • Tautologie: ' OR '1'='1 ist immer wahr.
  • Bedingung gilt für jeden Datensatz der Tabelle
  • Loggt oftmals ersten Treffer ein (häufig Administrator)
Login-Formular (TAUTOLOGIE MODE)
SELECT * FROM users WHERE username = '' AND password = ''

Die Werkzeugkiste

Arten von Angriffsvektoren:

  • 1. In-Band SQLi: Klassiker. Resultate direkt auf Webseite sichtbar.
  • 2. Blind SQLi: Keine Fehler sichtbar. Ja/Nein-Fragen (z. B.: „Dauert Antwort > 5s bei 'A'?“).
  • 3. Out-of-Band: Datenexfiltration über externe Kanäle (DNS).

UNION & Automatisierung

4. UNION Based SQLi: Kombiniert eingeschleuste Abfrage mit der originalen.

// 1. Original: SELECT name, preis FROM produkte WHERE id = '1'
// 2. Payload:  1' UNION SELECT user, pass FROM users --
// 3. Resultat: SELECT name, preis FROM produkte WHERE id = '1' UNION SELECT user, pass FROM users --'
  • Niemand testet mehr händisch: Tools wie sqlmap = Tausende Kombis/Sekunde.
  • Vollautomatisch (URL rein -> Datenbanktyp erkannt & Tabellen kopiert).
  • Erhöht Druck auf Entwickler enorm.

Fatale Konsequenzen

Es geht um Existenzen:

  • Finanzwelt: Kontostände manipulieren
  • Gesundheitswesen: Patientendaten löschen/stehlen
  • Extremfall: Remote Code Execution (OS-Zugriff)
Der IT-Super-GAU schlechthin.
Historische Fälle
  • Yahoo (2013): 3 Milliarden Konten kompromittiert.
  • Sony Pictures (2011): >1 Million Kundendaten. Verlust: $100+ Mio.

Die Lösung: Prepared Statements

Strikte Trennung von Code und Daten

  • Ablauf: Vorab SQL-Template senden: SELECT * FROM users WHERE user = ?
  • Datenbank kompiliert den strukturellen Befehl
  • Erst danach: Parameter als sturer Text übergeben
  • Angriff verpufft garantiert (Kein Filtern nötig!)
Login (SECURE PREPARED STATEMENTS)

Best Practices

Abwehrmechanismen in der Tiefe (Defense-in-Depth):

  • Input-Validierung: Nur valide Datentypen erlauben (z.B. Integer für Alter)
  • Least Privilege Prinzip: DB-User ohne globale Rechte betreiben
  • Sicheres Error Handling: SQL-Fehler komplett unterdrücken
  • WAF (Web App. Firewall): Bekannte Muster von vornherein blocken

Fazit

Sicherheit ist kein nachträgliches Feature, sondern das Software-Fundament.

  • Vermeidbares Relikt der IT-Geschichte
  • Verantwortung von uns Entwicklern!
  • Umgang mit Parametern entscheidet: Professionell vs. Tickende Zeitbombe
